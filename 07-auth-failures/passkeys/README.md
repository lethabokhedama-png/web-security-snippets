# Passkeys and WebAuthn

**OWASP:** A07 — Identification and Authentication Failures
**Risk Level:** Low (when implemented correctly — this is the most secure authentication available)
**Applies to:** Applications requiring phishing-resistant authentication

---

## What Are Passkeys?

Passkeys are a passwordless authentication standard based on WebAuthn (Web Authentication API) and FIDO2. Instead of a password, the user authenticates with a cryptographic key pair stored on their device — a phone, laptop, or hardware security key. The private key never leaves the device. The server only stores the public key.

Passkeys are the most secure widely available authentication method because they are:

**Phishing-resistant:** The key is bound to the exact domain it was registered with at a cryptographic level. A fake site at `my-googl.com` cannot receive a valid passkey response intended for `google.com`. Phishing attacks that work against passwords and even TOTP codes cannot work against passkeys.

**Breach-resistant:** No password to steal. The server stores only a public key, which is worthless without the corresponding private key on the user's device.

**Replay-resistant:** Each authentication uses a unique challenge generated by the server. Captured authentication responses cannot be replayed.

---

## The WebAuthn Protocol

**Registration:**
1. Server generates a random challenge and sends it with registration options (which authenticators to allow, user verification requirements)
2. The authenticator (device biometrics, hardware key) creates a new key pair, signs the challenge with the private key, and returns the public key and signature
3. Server verifies the signature and stores the public key alongside the user record

**Authentication:**
1. Server generates a fresh random challenge
2. The authenticator signs the challenge with the private key stored on the device
3. Server verifies the signature against the stored public key

The private key is never transmitted. The challenge ensures the response is fresh and cannot be replayed.

---

## Implementation Considerations

WebAuthn implementation is more complex than other authentication methods. A production implementation requires:

- A WebAuthn server library that handles credential verification, attestation, and challenge management
- A client-side JavaScript implementation using the Web Authentication API
- Storage for user credential records (credential ID, public key, sign count, user handle)
- Fallback authentication for users without a compatible device
- Account recovery when a passkey device is lost

Use a well-maintained WebAuthn library rather than implementing the protocol from scratch. The cryptographic operations and attestation verification are complex, and errors have serious security implications.
