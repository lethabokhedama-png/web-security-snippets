# Multi-Factor Authentication (MFA)

**OWASP:** A07 — Identification and Authentication Failures
**Risk Level:** Critical
**Applies to:** Any application where account compromise would cause significant harm

---

## What Is MFA and Why Does It Matter?

Multi-factor authentication requires users to present two or more independent credentials from different categories: something they know (password), something they have (phone, hardware key), or something they are (biometric). An attacker who obtains a user's password through a data breach, phishing, or credential stuffing cannot authenticate without also compromising the second factor.

According to Microsoft's security research, MFA blocks over 99.9% of automated account compromise attacks. Password breaches — which happen constantly, at scale — become largely irrelevant when MFA is enabled. The password is still necessary but no longer sufficient.

---

## MFA Types and Their Security Properties

**TOTP (Time-based One-Time Password):** A six-digit code generated by an authenticator app (Google Authenticator, Authy, 1Password, Bitwarden) using a shared secret and the current time. Codes change every 30 seconds. This is the most widely deployed form of MFA for web applications. It is resistant to replay attacks (the code expires quickly) but vulnerable to real-time phishing (an attacker who can trick a user into entering their code on a fake site within 30 seconds can relay it to the real site).

**SMS OTP:** A one-time code delivered via text message. More user-friendly than TOTP because it requires no app setup. Less secure because SMS can be intercepted through SIM swapping, SS7 vulnerabilities, and carrier-level compromise. Acceptable for most consumer applications but not recommended for high-security contexts.

**Email OTP:** Similar to SMS OTP but delivered via email. Security depends on the security of the user's email account. Appropriate for low-to-medium security applications.

**Hardware security keys (FIDO2/WebAuthn):** A physical device that generates a cryptographic response to a challenge. Completely phishing-resistant — the key is bound to the specific origin it was registered with and will not respond to phishing sites. The highest security form of MFA available. Covered in the passkeys/ folder.

**Push notifications:** A push notification sent to a registered mobile app. The user approves the login by tapping a button. Convenient but vulnerable to MFA fatigue attacks, where an attacker sends repeated push notifications until the user approves one out of frustration or confusion (the Uber breach in 2022 exploited this).

## Implementing TOTP

TOTP is standardised in RFC 6238 and is the most implementable form of MFA for most web applications. The flow is:

1. During setup: generate a random secret (typically 20 bytes, base32-encoded), store it encrypted against the user record, and display a QR code that authenticator apps scan to register the secret
2. During login: after password verification succeeds, prompt for the current TOTP code
3. Verification: use the TOTP library to verify the submitted code against the stored secret, accepting a small window (usually ±1 code period = ±30 seconds) to account for clock drift
4. Invalidate the code after use to prevent replay within the same 30-second window

---

## Backup Codes

Any MFA system must provide backup codes — single-use codes that allow recovery when the primary MFA device is unavailable. Generate 8–12 codes of 10–12 random characters each. Display them once during setup and instruct the user to store them safely. Hash them before storing (they are effectively passwords). Invalidate each code after a single use.

Without backup codes, a user who loses their authenticator device is permanently locked out of their account. With backup codes, they are at risk only if the backup codes are also compromised.

---

## MFA Fatigue Mitigation

If using push notification MFA, implement number matching — require the user to enter a number displayed on the login screen into the push notification interface. This proves the user is actively participating in the authentication and prevents attackers from sending notifications until one is accidentally approved.
